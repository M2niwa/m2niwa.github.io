Harness 部署策略选型：Rolling、Blue-Green 还是 Canary？

2026-06-29
Harness 支持多种部署策略，但该选哪个？深入对比 Basic、Rolling、Blue-Green 和 Canary 四种策略的原理、适用场景和风险差异，附实战选型决策树。

上篇讲了怎么在 Harness 中搭第一条 Pipeline——但 Pipeline 跑起来只是第一步。真正决定部署质量的是你选的部署策略。用错了策略，再好的 Pipeline 配置也可能在上线时翻车。

Harness 的部署策略定义了新版本如何替换旧版本。它原生支持四种策略：Basic、Rolling、Blue-Green 和 Canary。每种有不同的风险模型和适用场景。

Basic：全量替换，简单但不安全

Basic 部署是最直接的方式——所有节点（pods、实例等）在同一时间被新版本替换。没有渐进过程，没有验证窗口，一刀切。

优点：配置最简单，部署速度最快。缺点也很明显：如果新版本有 bug，所有用户同时受影响，而且回滚需要完整重新部署旧版本——这个过程中服务可能完全不可用。

Harness 官方文档明确说 Basic 部署「不适用于生产环境」。它的定位是给你做实验和快速原型用的。如果你只是在开发环境里玩玩，Basic 够用；但只要涉及真实用户，往上再看。

Rolling：逐个替换，零 downtime 但慢

Rolling 部署的策略是：所有节点被逐个（或按批次）替换。每替换一个节点后等待它健康检查通过，再替换下一个。

好处是零 downtime——始终有节点在服务中。坏处是部署窗口长（节点越多越慢），而且新旧版本会短暂共存——这对 API 向后兼容性有要求。如果你的新版本改了数据库 schema 或不兼容旧 API，Rolling 部署期间会出现奇怪的问题。

Rolling 在 Harness 中常作为 Canary 部署的前置阶段——先在 QA 环境做 Rolling 部署验证，通过后再在 Production 做 Canary。

Blue-Green：即时切换，但成本翻倍

Blue-Green 的核心思想是维护两套完全相同的环境——Blue（当前生产版本）和 Green（新版本）。QA 在 Green 环境完成验证后，通过负载均衡器一次性将全部流量从 Blue 切到 Green。

优势是切换几乎是瞬时的，回滚也极其简单——把流量切回 Blue 就行。代价是基础设施成本翻倍——你需要两套完整的生产环境。对于资源敏感的小团队或个人项目，这个成本可能难以接受。

有些厂商把 Blue-Green 叫 Red-Black 部署，本质是一样的。

Canary：渐进式验证，最安全但最复杂

Canary 部署是目前最推荐的生产部署策略——新版本先部署到一小部分实例（如 5%），观察一段时间后逐步扩大比例：10% → 25% → 50% → 100%。每个阶段都有验证门禁——如果监控指标异常，自动停止并回滚。

Canary 的核心优势是把爆炸半径控制在最小。即使新版本有严重 bug，最坏情况也只影响 5% 的用户，而不是全部。Harness 的 Canary 实现还有一个细节：在 Kubernetes 上，Canary 阶段会使用额外实例而非替换已有实例，保证生产版本不受影响。

代价是 Pipeline 配置更复杂，部署时间更长（需要多个验证阶段）。但对比一次全量部署翻车带来的损失，这点复杂度是值得的投资。

怎么选？一个简易决策树

  • 你在开发/测试环境 → Basic 或 Rolling
  • 生产环境，但团队小、资源有限 → Rolling
  • 生产环境，能接受双倍基础设施成本 → Blue-Green
  • 生产环境，追求最小风险 → Canary

我个人网站的部署选择了 Rolling——因为只有一台服务器，Blue-Green 成本太高，Canary 又太复杂。但对于团队协作的商业项目，Canary 几乎是最优解。
