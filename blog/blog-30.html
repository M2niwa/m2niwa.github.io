Harness Feature Flags：让部署和发布彻底解耦

2026-07-12
为什么「部署了」不等于「上线了」？深入 Harness Feature Flags 的工作原理——SDK 接入、GitOps 配置、与 Canary 策略的配合，以及将发布风险从「全有或全无」变为「渐进可控」的完整实践。

上一篇讨论了部署策略的选型——但即使用了 Canary 部署，仍然有一个根本问题没解决：部署和发布是绑在一起的。代码推到生产环境的那一刻，功能就对用户可见了。你没法先把代码部署上去，等确认没有问题后再「打开」功能。

这就是 Harness Feature Flags 要解决的问题。

部署 ≠ 发布

传统 CI/CD 流程中，「部署」和「发布」是同一件事——代码部署到生产环境 = 功能对用户可见。Feature Flags 的核心思路是把这两个概念彻底解耦。

用 Feature Flags 后的流程变成了：

  1. 部署：把包含新功能的代码推到生产环境，但功能默认关闭
  2. 验证：在真实生产环境中测试（开启 flag 给内部用户或 1% 用户）
  3. 发布：确认无问题后，逐步扩大 flag 的开启范围

这带来的变化是根本性的——部署不再是一次心惊胆战的赌博。即使部署的代码有 bug，只要 flag 关着，用户完全不受影响。修复也不需要重新部署——改代码、部署、再开 flag 就行。

Harness Feature Flags 的工作方式

Harness 的 Feature Flags 由三个组件构成：

  • Flag 管理端：在 Harness 平台上创建、配置、管理 flag，支持 GitOps（flag 配置存在 Git 仓库中）
  • SDK：客户端库，嵌入你的应用代码中。所有 SDK 都是开源的，按语言提供（Java、Python、Go、Node.js、React 等）
  • 评估引擎：运行时判断当前请求是否应该看到新功能

接入 SDK 的代码非常简单——本质上就是一个条件判断：

// 用 Harness SDK 包裹新功能
import { useFeatureFlag } from '@harnessio/ff-react-client-sdk';

function Dashboard() {
  const showNewUI = useFeatureFlag('new_dashboard_ui');

  return showNewUI ? <NewDashboard /> : <OldDashboard />;
}

Flag 的开启/关闭不需要修改代码或重新部署——在 Harness 控制台点一下按钮，SDK 在几秒内就能感知到变化并更新行为。

和 Canary 部署的配合

Feature Flags + Canary 部署是一对天然的组合。Canary 控制有多少实例跑新版本（基础设施层面），Feature Flags 控制新版本中哪些功能对用户可见（应用层面）。

一个典型的协同流程：

  1. Canary 部署到 5% 实例 → 开启 flag 给 1% 用户 → 监控 10 分钟
  2. 扩大到 25% 实例 → flag 开到 10% 用户 → 继续监控
  3. 100% 实例部署完成 → flag 逐步开到 100% 用户

如果在任何阶段发现问题：关闭 flag 即可——比完整的 Canary 回滚快得多（秒级 vs 分钟级）。

对团队协作的影响

Feature Flags 不只是技术工具，它改变了团队的协作模式。Harness 文档里提到了几个立竿见影的变化：

  • 产品经理可以独立控制发布节奏——不需要等开发重新部署
  • QA 可以在生产环境做真实测试——flag 只对内部用户开放
  • 新成员入职更快——不需要理解整个发布流程，改 flag 就行

代价是需要管理越来越多的 flag（flag 膨胀）。如果没有定期清理机制，代码里会堆积大量已废弃的分支判断——这需要在流程中建立 flag 生命周期管理（创建 → 全量发布 → 移除 flag 代码）。

但总体来看，Feature Flags 带来的风险降低和发布灵活性，远超管理成本。尤其在 Harness 这样把 FF 和 CD 深度整合的平台上，它不是一个「附加功能」，而是整个交付体系的有机组成部分。
