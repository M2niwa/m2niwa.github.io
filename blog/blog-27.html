从手写脚本到 Harness CD：第一条 Pipeline 的搭建实录

2026-06-02
把个人项目的部署从手写 bash 脚本迁移到 Harness CD 的全过程——Pipeline 的 Stage/Step 层级结构、Webhook 触发器配置、Delegate 代理部署，以及第一次自动部署中踩的坑。

在接触 Harness 之前，我的部署流程就是 SSH 进服务器 → git pull → docker-compose up -d --build → 祈祷。这种「手工部署」用在小项目上还能忍，但一旦需要管理多个环境（开发/测试/生产），手动操作的出错概率和心智负担就直线上升。

Harness 是一个完整的软件交付平台，最初以 CD（Continuous Delivery）闻名，现在扩展到 CI、Feature Flags、云成本管理、安全测试等多个模块。这篇文章聚焦在最核心的 CD 部分——怎么从零搭第一条能自动部署的 Pipeline。

Harness CD 的核心抽象：Pipeline → Stage → Step

理解 Harness 最关键的是掌握它的层级结构：

  • Pipeline：一次完整的交付流程。一个 Pipeline 可以包含多个 Stage。
  • Stage：按环境或阶段划分的执行单元。比如 Dev Stage → Staging Stage → Production Stage。
  • Step：最小操作单元——执行脚本、发送通知、等待审批等。

这种三层抽象和你熟悉的 CI/CD 工具（GitHub Actions、GitLab CI）的理念类似，但 Harness 把 Stage 的概念提升到了一等公民——每个 Stage 可以独立配置部署策略、审批流程和失败处理。

配置第一条 Pipeline

我用 Harness 的 YAML 编辑器直接写配置（Harness 也提供可视化编辑器，但源码控制场景下 YAML 更方便）：

pipeline:
  name: personal-website-deploy
  identifier: personal_website_deploy
  projectIdentifier: personal_projects
  orgIdentifier: default
  stages:
    - stage:
        name: Deploy
        type: Deployment
        spec:
          serviceConfig:
            serviceRef: website
          execution:
            steps:
              - step:
                  type: ShellScript
                  name: Pull and Build
                  spec:
                    shell: Bash
                    onDelegate: true
                    source:
                      type: Inline
                      spec:
                        script: |
                          git clone https://github.com/M2niwa/m2niwa.github.io
                          # build commands

这里有个关键参数：onDelegate: true。这意味着脚本在你自己的 Harness Delegate 上执行，而不是在 Harness 云端沙箱。对于需要访问私有仓库或内网服务器的场景，这几乎是必须的。

Delegate：连接 Harness 和你服务器的桥梁

Delegate 是 Harness 架构中最重要的组件——它是一个运行在你基础设施（Kubernetes、Docker、VM）上的轻量级代理，负责：

  • 从 Harness 平台拉取 Pipeline 任务
  • 在你的环境中执行 Shell 脚本、kubectl 命令、Terraform 等
  • 将执行日志和状态回传到 Harness 平台

我用了 Docker 部署 Delegate，只需一个 docker run 命令就能启动。但这里有一个易错点：Delegate 不会自动更新。如果 Harness 平台升级了新功能，旧版 Delegate 可能不支持新的 Step 类型，导致 Pipeline 执行报错。解决方案是定期 pull 最新镜像或使用 Kubernetes 的自动更新策略。

接入 GitHub Webhook 触发器

配好 Pipeline 后，下一步是让它「自动运行」——每次推送代码到主分支时触发部署。Harness 原生支持 Webhook 触发：

  1. 在 Harness 中创建 Trigger，选择 Webhook 类型
  2. 复制生成的 Webhook URL 和 Secret
  3. 在 GitHub 仓库的 Settings → Webhooks 中填入

配置完成后，整个部署链路变成：

git push → GitHub Webhook → Harness Trigger → Pipeline 执行 → 部署完成

从 push 到线上生效大约 2-3 分钟。第一次看到这个流程跑通的感觉确实不错——部署不再是需要我亲自盯着的事。

不过 Harness 的入门还是有一些上手成本。YAML 配置的选项非常多，文档虽然全面但初次阅读时容易迷失。建议从官方 Tutorial 开始，跟着一步步操作，而不是从零看文档。
